<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| app runtime → Postgres | Pooler Supavisor é a ÚNICA entrada legítima em runtime; acesso direto (5432) é privilegiado |
| Prisma migrate → Postgres | Usa DIRECT_URL (5432) apenas temporariamente — credenciais devem ter permissão de DDL |
| webhook MP futuro → Order | `Order.paymentId UNIQUE` é o guard matemático contra replay/duplicação (Pitfall #1) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-04 | D (Denial of Service) | Pool de conexões Supabase | mitigate | DATABASE_URL com `connection_limit=1` + pooler 6543 — padrão oficial Supabase+Prisma+Vercel |
| T-00-05 | T (Tampering) | Idempotência de pedido pago | mitigate | `Order.paymentId @unique` no schema — garante que 2 webhooks com mesmo paymentId não criem 2 pedidos (base do Pitfall #1, endereçado de verdade na Fase 4 mas fundação é aqui) |
| T-00-06 | I (Information Disclosure) | Credenciais DB | mitigate | .env.local no .gitignore (Plan 01) + senha forte gerada pelo Supabase |
| T-00-07 | S (Spoofing) | Role escalation | mitigate | enum `Role { CUSTOMER, ADMIN }` — única forma de virar ADMIN é UPDATE manual no DB (Plan 03 documenta) |
</threat_model>
